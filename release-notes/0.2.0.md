**Headline: DuckDB joins Neo4j as a first-class backend.** Pick at startup with
`--backend duckdb`. Same mapper, same invariants, single-file `.duckdb` output
that anyone can `SELECT cost_per_agent FROM ops` against — no graph database
required.

## What's new

### Backends are pluggable

- **New `Sink` protocol** — both `Neo4jSink` and the new `DuckDBSink` satisfy it structurally. Choose your backend at startup; everything else is identical.
- **`make_sink(config)` factory** — `Neo4jConfig(...)` and `DuckDBConfig(path=...)` as tagged-union configs. `config_from_env` resolves CLI flags + env vars + defaults.
- **Backend selection** — `--backend {neo4j,duckdb}` flag on the loader CLI, or `OTGG_BACKEND` env var. Defaults to `neo4j` (preserves prior behaviour).
- **`GenAIExporter`** is the new generic OTel SpanExporter. `Neo4jGenAIExporter` is retained as an alias subclass for back-compat — no breaking import changes.

### DuckDB backend — analytics-shaped

- Wide **`ops`** table (one row per Operation) with structural edges flattened as FK columns: `session_id`, `agent_id`, `model_provider`, `model_name`, `tool_name`, `data_source_id`, `parent_span_id`, `service_name`. `INSERT … ON CONFLICT … DO UPDATE` for idempotency.
- Dimension tables: `sessions`, `agents`, `models`, `tools`, `data_sources`, `resources`, plus `agent_delegations` for the agent → agent edge.
- **Parallel SQL saved-query library** (`saved_queries_sql.py`) — 12 named queries including a recursive-CTE `cost_by_agent_with_descendants` that walks the delegation graph.
- **Air-gapped friendly** — DuckDB is in-process. Single-file output you can `scp`, email, or hand to a stakeholder.

### `Resource` is no longer aspirational

- Mapper now emits a **`Resource`** node from `resourceSpans[*].resource.service.name`, closing the v0.1 known limitation.
- **`Operation.service_name`** is denormalised onto every operation as the natural FK to `Resource`. Lights up "cost per service" as a no-join SQL query and as a queryable property in Neo4j.

## Install

Backends are now optional extras:

```bash
pip install 'otel-genai-graph[neo4j]'    # graph backend
pip install 'otel-genai-graph[duckdb]'   # SQL analytical backend
pip install 'otel-genai-graph[all]'      # both, for tooling that switches
```

## Quickstart — DuckDB

```bash
python -m otel_genai_graph.load tests/fixtures/*.json \
    --backend duckdb --duckdb-path ./trace.duckdb

duckdb ./trace.duckdb \
    "SELECT model_provider, model_name, count(*) AS calls,
            sum(coalesce(input_tokens,0)+coalesce(output_tokens,0)) AS tokens
     FROM ops WHERE model_provider IS NOT NULL
     GROUP BY 1,2 ORDER BY tokens DESC"
```

## Breaking changes

- **`neo4j` is now an optional extra.** Bare `pip install otel-genai-graph` no longer pulls the Neo4j driver. Add `[neo4j]` (or `[all]`) if you used the Neo4j backend. Existing `pip install -e ".[dev]"` workflow unchanged (dev pulls both).
- **`Resource` nodes are now emitted by the mapper.** Queries that count *total* nodes (e.g., `MATCH (n) RETURN count(n)`) will see N+1 results per resource bundle. Per-label queries are unaffected; this only matters for naïve totals.

## Compatibility

- Python 3.10 / 3.11 / 3.12 / 3.13.
- Neo4j 5.x (unchanged).
- DuckDB ≥ 1.0 (new).
- OpenTelemetry SDK ≥ 1.27.

## Asymmetry between backends — by design

| Capability | Neo4j | DuckDB |
|---|---|---|
| Graph traversals (`MATCH p=...`) | ✅ | ❌ |
| SQL aggregates over wide rows | via Cypher | ✅ idiomatic |
| Interactive HTML graph viz (`tools/render_graph.py`) | ✅ | ❌ |
| Single-file output | ❌ | ✅ |
| In-process, no server | ❌ | ✅ |

Use Neo4j when you want graph queries and visual exploration; use DuckDB
when you want SQL analytics, CI gates, or air-gapped portability. See
[README → Backends](https://github.com/kums1234/otel_genai_graph_exporter/blob/v0.2.0/README.md#backends--neo4j-or-duckdb).
